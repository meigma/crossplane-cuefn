# DESIGN — `cuefn check`: static module health for cuefn modules

Status: draft for developer review (session 011).
Companion to `cuefn test` (session 010): `check` is the instance-free static
gate, `test` is the instance-driven behavioral gate.

## 1. Goal

Give module authors one command that answers "is this module statically
healthy?" without needing an XR instance: formatting is canonical, the CUE
evaluates cleanly, the XRD generates (structural, no forbidden disjunctions),
and — optionally — the generated XRD matches a reviewed golden. It replaces
the hand-rolled scripts every serious author otherwise writes (this repo's
own `root:xrd-check` moon task is exactly that script).

The three-verb split, which docs must state crisply:

| Verb | Question | Needs an instance? |
|---|---|---|
| `cuefn check` | Is the module statically healthy? Does its XRD generate (and match the golden)? | No |
| `cuefn test` | Does the module render the right resources for these inputs? | Yes (txtar cases) |
| `cuefn validate` | Is this one concrete XR accepted by the module's schema? | Yes (one XR) |

Mirrors `go vet` / `go test`.

### Non-goals

- **No `cue mod tidy`.** Verified against cuelang.org/go v0.16.1: tidy lives
  in `internal/mod/modload`; the public `mod/*` packages expose no tidy API.
  Shelling out to a `cue` binary would break the single-binary promise. Docs
  point at `cue mod tidy --check` as the complement; revisit if upstream
  exports it.
- **No behavior/render checks** — that is `cuefn test`.
- **No `--fix`/auto-format in v1.** Formatting failures name the files;
  authors run `cue fmt`. `--update` never rewrites hand-written source (the
  session-010 rule); a later `--fix` flag can format in place if wanted.
- **No OCI-ref checking in v1** — local `--dir` only, same as `test`. The
  OCI-ref fast-follow is shared with `test` (both fall out of `moduleLoader`
  nearly free) and should land for both at once.
- **No subpackage sweep.** v1 vets the root module instance (what the engine
  and codegen actually load), not `./...` over arbitrary subpackages; cuefn
  modules are single-package by contract.

## 2. The checks (fixed order, all reported)

Run all checks and report each; do not stop at the first failure
(`--fail-fast`? no — there are only three units; always run all).

### 2.1 `fmt` — canonical formatting

Every `.cue` file under `--dir` (recursive), including `cue.mod/module.cue`,
excluding the legacy vendoring dirs `cue.mod/{pkg,gen,usr}` and hidden
directories. Each file's bytes are compared against
`cuelang.org/go/cue/format` output (default options — parity with bare
`cue fmt`; pinned by fixture tests, not assumed). Failure lists the
unformatted files. Note: the repo itself currently has NO CUE formatting
gate (`root:format` is golangci-lint only) — this check closes that hole for
the example module too.

### 2.2 `vet` — the module evaluates cleanly

Load the module through the same `moduleLoader(dir, cacheDir)` path as
render/generate/test (dependency-aware, central registry default), then
non-concrete validate — the programmatic equivalent of the documented author
check `cue vet -c=false ./...`, encoding the session-006 footgun (bare
`cue vet` fails on required-no-default fields). Exact `Validate` options are
an implementation detail pinned by two fixtures: a known-bad module must
fail, the hermetic good module must pass. Because evaluation includes the
contract import's unification, closed-contract violations (e.g.
`out.resorces`) surface here with `cueerr.Wrap`-quality messages.

### 2.3 `xrd` — the XRD generates; optionally matches a golden

Always: `schema.GenerateXRDYAML(module)` must succeed. This inherits the
whole existing quality bar for free — definitions-only reduction, `$ref`
inlining, materialized defaults, the type-crossing-disjunction rejection
naming the field, and the `NewStructural`/`ValidateStructural` self-check.
This is the failure surface render tests cannot see.

With `--xrd <path>`: additionally compare the generated YAML against the
golden file, with the `test`-command golden lifecycle:

- **Compare:** byte-exact after stripping leading full-line `#` comments
  from both sides. Rationale: the machine-owned header (below) must not
  break files previously written headerless by `cuefn generate --output`,
  and the existing `example/xrd.yaml` passes unchanged on day one. Mismatch
  reports the shared LCS line diff (same renderer as `want.yaml` failures).
- **Seed:** `--xrd` given, file absent → write it (with header), report
  `SEED`, exit non-zero ("review and commit"). Refused in CI mode.
- **Update:** `--update` rewrites a drifted golden (with header), reports
  `UPDATE`. Refused in CI mode (same error text pattern as `test`).
- **Header:** seeded/updated files begin
  `# Generated by cuefn check; DO NOT EDIT. Re-bless with: cuefn check --xrd <path> --update`
  — closing the Phase-8 follow-up (example/xrd.yaml lost its generated
  header; exact-match xrd-check couldn't tolerate one).

No `--xrd` → no golden semantics at all (generate-succeeds only). No
convention path in v1 (an implicit `<dir>/xrd.yaml` would get packaged by
`cue mod publish`, which packages everything — an opt-in flag avoids
surprise bloat; revisit if demand appears).

## 3. CLI specification

```
cuefn check [--dir <module-dir>] [--cache-dir <dir>] [--xrd <golden-file>] [--update] [--ci]
```

- `--dir` default `.` (like `test`; no positional module-ref — local only).
- `--cache-dir` as on render/generate/validate/test.
- `--ci` refuses seeding and `--update`; auto-enabled by the same truthy-CI
  env detection (`envCI()`), reusing `test`'s exact semantics and wording.
- Report vocabulary is `test`'s, one line per unit:

```
PASS fmt
PASS vet
FAIL xrd
    <indented detail: generate error, or the LCS golden diff>

2 passed, 1 failed, 0 seeded, 0 updated
```

- Exit non-zero on any FAIL or SEED (summary error mirrors `summarize()`).
- Cobra Long text must state the check/test/validate delineation in one
  sentence each.

## 4. Architecture (hexagonal)

- **`internal/textdiff`** — new tiny package: `diffLines` moves here
  verbatim from `internal/testharness/diff.go` (exported as `Lines`);
  testharness re-uses it. Behavior-preserving extraction, same pattern as
  the #69 snapshot extraction.
- **`internal/check`** — the core, bytes/values in, findings out; no file
  discovery, no printing:
  - `Fmt(files map[string][]byte) []string` (unformatted file names, sorted)
    — or per-file `(name, formatted bool)`; exact shape at implementation.
  - `Vet(module cue.Value) error`.
  - `XRD(module cue.Value, golden []byte, haveGolden bool) (generated []byte, diff string, err error)`
    — comment-stripping compare + `textdiff.Lines` live here.
  - Depends only on `internal/render` (nothing — takes loaded values),
    `internal/schema`, `internal/cueerr`, `internal/textdiff`, and
    `cue/format`. No cobra, no os.
- **`internal/cli/check.go`** — thin adapter: flag parsing, file walking and
  reads, `moduleLoader` + `render.LoadModule`, golden read/seed/update
  writes, PASS/FAIL/SEED/UPDATE printing via `options.Out`, exit mapping.
  File-walk exclusions (`cue.mod/{pkg,gen,usr}`, hidden dirs) live here.

`internal/check` must not import `internal/testharness` or vice versa; the
shared diff is why `textdiff` exists.

## 5. Dogfooding and gates

- Replace the `root:xrd-check` moon task body with
  `go run ./cmd/cuefn check --dir example/module --xrd example/xrd.yaml --ci`
  (keep the task name; it now also fmt+vets the example). Its comment block
  updates the re-bless instruction to `... check --xrd ... --update`.
- `example/xrd.yaml` passes unchanged at first (comment-stripping compare);
  re-bless once with `--update` in the PR so it gains the header — the
  committed diff is the header lines only.
- Unit fixtures in `internal/check`/`internal/cli` testdata: an unformatted
  file, a vet-failing module, a type-crossing-disjunction module, golden
  match/drift/seed cases. All offline (hermetic module pattern) — no new
  integration or e2e infrastructure.
- CI needs no new job: the blocking `check` gate already runs `xrd-check`,
  and CI already warms the CUE module cache for the example's central-
  registry dep.

## 8. Open questions (developer input welcome, defaults chosen)

1. **Golden compare tolerance** — default: strip leading full-line comments,
   else byte-exact. Alternative rejected: full comment-tolerance everywhere
   (weakens the guarantee) and pure byte-exact (breaks existing headerless
   goldens and forbids the header).
2. **Should `cuefn generate --output` also write the header?** Default: no —
   generate's output is the artifact you `kubectl apply`; check owns the
   golden. Revisit if header drift annoys.
3. **Unit naming** — `fmt` / `vet` / `xrd`. Alternative: `generate` instead
   of `xrd`; `xrd` chosen because the unit covers generate+golden together.

## 9. Key facts this design leans on

- `cue/format` (`format.Source`) is public in v0.16.1 → in-process fmt.
- Tidy is not importable (`internal/mod/modload`) → excluded, documented.
- `schema.GenerateXRDYAML` already carries the structural self-check and
  disjunction guard → the `xrd` unit is a thin wrapper.
- `test.go`'s lifecycle mechanics (`envCI`, seed/update refusal wording,
  PASS/FAIL/SEED/UPDATE, `summarize`, `indent`) are the UX contract to
  mirror, not re-invent.
- `root:xrd-check` today: bash `diff -u` against `bin/xrd.gen.yaml` — the
  bespoke script this command deletes.
- `root:format` is Go-only; no CUE fmt gate exists in the repo today.
